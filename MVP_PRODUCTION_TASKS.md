## 1. Security

### SEC-01 — Gemini API key exposed in source code
**Priority: P0 | Effort: M**

`lib/utils/constants.dart` line 3 contains a live Gemini API key hardcoded as a Dart string constant (`AIzaSyD_ibdZ3o5M_rUy6FhY0GK_cpCVgWo6Sa0`). Any user who reverse-engineers the APK or reads the source can extract and abuse it. The key is also committed to version history permanently.

**Action:** Remove the key from `constants.dart`. Proxy all Gemini calls through a Cloud Function or Firebase App Check-protected backend endpoint so the key never ships in the client binary. As an interim measure, restrict the key in Google Cloud Console to only the app's SHA-256 fingerprint (Android) and bundle ID (iOS), and set a per-day quota cap.

---

### SEC-02 — Firebase API keys exposed in tracked files
**Priority: P0 | Effort: S**

`lib/firebase_options.dart` and `android/app/google-services.json` both contain Firebase API keys in plain text committed to the repository. The Android key (`AIzaSyDnPlfaSHwLPjy84anU7qJOvb0O49J35M8`) appears verbatim in both files.

**Action:** Firebase client keys are not secret by themselves, but they must be protected by Firebase Security Rules (see SEC-05). Add `google-services.json` and `GoogleService-Info.plist` to `.gitignore` immediately and re-generate them from the Firebase console for distribution builds. Verify Firebase App Check is enabled so that only signed builds of the app can access project resources.

---

### SEC-03 — RevenueCat keys are placeholder strings — paywall is a mock
**Priority: P0 | Effort: M**

`lib/utils/constants.dart` lines 9–10 have `'<YOUR_REVENUECAT_IOS_KEY>'` and `'<YOUR_REVENUECAT_ANDROID_KEY>'`. The `_purchase()` method in `lib/screens/paywall/paywall_screen.dart` (lines 56–76) is entirely commented out and replaced with a mock that sets `isPro = true` unconditionally after 800ms. Any user who taps "Start Free Trial" gets Pro for free.

**Action:** Integrate `purchases_flutter` SDK properly. Configure both platform keys in constants. Implement the commented-out purchase flow, including entitlement verification and `Purchases.restorePurchases()` on the "Restore" button (currently a TODO on line 302).

---

### SEC-04 — Scan quota enforcement is client-side only
**Priority: P0 | Effort: L**

The free-tier scan limit (`maxFreeScansPerDay = 3`) is enforced exclusively in `lib/models/user_profile.dart` (`canScan` getter, line 65) and `lib/screens/scan/scan_screen.dart` (lines 71–79). A user can bypass it by modifying the `dailyScanCount` field in their own Firestore document, or by calling the Gemini endpoint directly using the extracted key (see SEC-01).

**Action:** Move quota enforcement to a Cloud Function that counts scan events per UID per UTC day before calling Gemini. The client check can remain as a UX guard, but it must not be the only gate.

---

### SEC-05 — Firestore security rules are unverified / likely default
**Priority: P0 | Effort: M**

No `firestore.rules` file exists in the repository. Firebase default rules for new projects allow no reads or writes, but projects in "test mode" allow everything. The data model (`users/{uid}/meal_logs`, `users/{uid}/scan_events`) is structured correctly for per-user rules, but those rules must be authored and deployed.

**Action:** Create `firestore.rules` at the repo root. At minimum: allow read/write only when `request.auth.uid == resource.data.userId` for meal_logs, and `request.auth != null && request.auth.uid == uid` for the user document path. Deploy via `firebase deploy --only firestore:rules`.

---

### SEC-06 — Anonymous auth fallback writes data under `local_user` UID
**Priority: P1 | Effort: S**

`lib/services/auth_service.dart` lines 15–17 catch anonymous sign-in failures and fall back to the hardcoded string `'local_user'`. If multiple devices hit this fallback (e.g., anonymous auth is disabled in the console), their data merges under a single document. This is also a data integrity issue.

**Action:** Remove the `local_user` fallback. If anonymous auth is unavailable, surface a proper error to the user rather than silently proceeding with a shared UID.

---

### SEC-07 — No Firebase App Check configured
**Priority: P1 | Effort: M**

There is no App Check setup in `main.dart` or anywhere in the project. Without App Check, any script with the Firebase project credentials can read/write Firestore directly, bypassing security rules.

**Action:** Enable App Check with Play Integrity (Android) and DeviceCheck (iOS) in the Firebase console. Add `FirebaseAppCheck.instance.activate(...)` before `Firebase.initializeApp()` in `lib/main.dart`.

---

### SEC-08 — No input validation on numeric fields in manual entry
**Priority: P1 | Effort: S**

`lib/screens/diary/add_food_screen.dart` `_isValid` getter (lines 57–60) only checks that `_calCtrl.text` parses as a double. Values like `-9999` calories or `99999` grams of protein are accepted silently and written to Firestore. The edit dialog in `lib/screens/scan/scan_screen.dart` (lines 219–228) has the same gap.

**Action:** Add range validation: calories 0–5000, macros 0–500g each, serving 1–5000g. Show an inline error on out-of-range submission.

---

### SEC-09 — `debugPrint` logs full Gemini API response body in production builds
**Priority: P1 | Effort: S**

`lib/services/ai_scan_service.dart` lines 63–65 and 78–79 call `debugPrint` with the full HTTP response body and parsed food item names. `debugPrint` is no-op in release mode by default, but if `flutter run --release` is replaced with `flutter run --profile` or a custom build, private nutritional data appears in device logs.

**Action:** Replace all `debugPrint` calls in service files with a conditional logger (`kDebugMode` guard) or remove them entirely from `ai_scan_service.dart`.

## 2. Performance & Optimization

### PERF-01 — Full image bytes sent base64-encoded without compression
**Priority: P1 | Effort: S**

`lib/services/ai_scan_service.dart` line 13 reads the entire image file into memory as bytes and base64-encodes it. `image_picker` is configured with `imageQuality: 92, maxWidth: 1600` in `scan_screen.dart` line 82, but a 1600px-wide JPEG at 92% quality is still 500 KB–1.5 MB. Base64 adds ~33% overhead, so Gemini payloads routinely exceed 2 MB.

**Action:** Reduce `imageQuality` to 75 and `maxWidth` to 1024. Add a size cap: if `bytes.length > 512 * 1024`, resize using the `image` package (already in `pubspec.yaml`) before encoding. This will cut latency by 40–60% and reduce Gemini token costs.

---

### PERF-02 — `getMealsForDate` uses ISO 8601 string comparison in Firestore
**Priority: P1 | Effort: M**

`lib/services/database_service.dart` lines 49–51 query `dateTime` using `isGreaterThanOrEqualTo` / `isLessThan` on ISO 8601 strings. String comparison works accidentally for ISO dates in the same timezone, but breaks across DST boundaries and for users in negative-offset timezones. It also prevents Firestore from using a composite index efficiently.

**Action:** Change `dateTime` storage to a Firestore `Timestamp` type. Update `MealLog.toMap()` to use `Timestamp.fromDate(dateTime)` and `fromMap()` to call `.toDate()`. This also requires a `firestore.indexes.json` entry for `(dateTime ASC)` on the `meal_logs` collection.

---

### PERF-03 — Dashboard fires 7 separate Firestore reads on every render
**Priority: P1 | Effort: M**

`lib/screens/dashboard/dashboard_screen.dart` `_loadWeeklyData()` (lines 42–57) loops over 7 days and calls `diary.getMealsForDateRaw()` sequentially inside a `for` loop. Each call results in a Firestore `getDocs` network round-trip. With offline persistence this is mitigated, but on first load or after cache expiry this creates 7 sequential fetches.

**Action:** Run the 7 futures in parallel with `Future.wait`. Better: add a Firestore query that fetches all meals for the last 7 days in a single query and groups them by date client-side. This reduces latency from 7× round-trip to 1×.

---

### PERF-04 — `DatabaseService` instantiated as `new DatabaseService()` inside `_ScanScreenState._confirm()`
**Priority: P2 | Effort: S**

`lib/screens/scan/scan_screen.dart` line 121 calls `final db = DatabaseService()` inside an async handler, creating a new service object on every confirmation rather than reusing the singleton injected via the `Provider` tree. This is a minor inconsistency that could cause double-initialization problems if `DatabaseService` ever acquires constructor-time side effects.

**Action:** Pass `DatabaseService` through the Provider tree (it is already available via `DiaryProvider`) and use `context.read<DiaryProvider>()._db` or expose a top-level provider.

---

### PERF-05 — `CalorieSummaryCard` re-runs 1200ms animation on every calorie change
**Priority: P2 | Effort: S**

`lib/widgets/calorie_summary_card.dart` `didUpdateWidget` (lines 47–50) restarts the full 1200ms fill animation every time `calories` changes. If multiple meals are added in quick succession (e.g., AI scan returns 4 items at once), the ring repeatedly restarts from zero. This is jarring UX and causes unnecessary repaints.

**Action:** Use a short 300ms transition animation rather than a full ring fill restart on subsequent updates. Only run the full 1200ms animation on the very first load.

## 3. Error Handling & Resilience

### ERR-01 — Gemini network failure shows raw exception string to users
**Priority: P1 | Effort: S**

`lib/screens/scan/scan_screen.dart` line 107 sets `_error = 'Analysis failed: $e'` where `$e` is the raw Dart exception. This can expose internal details like `SocketException: Failed host lookup: 'generativelanguage.googleapis.com'` or the full Gemini error JSON to users.

**Action:** Map common exceptions to user-friendly Hebrew messages: network timeout → "אין חיבור לאינטרנט. נסה שוב.", HTTP 429 → "מגבלת הסריקה היומית הושגה.", HTTP 5xx → "שגיאת שרת. נסה שוב מאוחר יותר.", generic → "הניתוח נכשל. צלם תמונה ברורה יותר."

---

### ERR-02 — Gemini response parsing has no defensive guards
**Priority: P1 | Effort: S**

`lib/services/ai_scan_service.dart` lines 73–81 access `data['candidates'][0]` and `candidate['content']['parts'][0]['text']` without null checks. If Gemini returns an empty `candidates` array (which happens on safety filter blocks), the app throws a `RangeError` that manifests as an unhandled exception.

**Action:** Wrap the response parsing in try/catch. Check `candidates.isNotEmpty` before indexing. Check `finishReason`: if it equals `'SAFETY'` or `'RECITATION'`, throw a typed `AIScanException` with a user-visible message.

---

### ERR-03 — `UserProvider.load()` silently swallows all errors
**Priority: P1 | Effort: S**

`lib/providers/user_provider.dart` lines 16–21: the `catch (_)` block sets `_user = null` and continues. If Firestore is unavailable during the first load (e.g., no network and no cached data), the app silently routes to `OnboardingScreen` and the user re-onboards, losing their existing profile.

**Action:** Distinguish between "user document does not exist" (new user → onboarding) and "Firestore error" (network failure → show retry screen). Expose an `error` state in `UserProvider` and handle it in `_AppEntryState`.

---

### ERR-04 — No retry mechanism for failed AI scans
**Priority: P1 | Effort: S**

After a failed AI scan, `lib/screens/scan/scan_screen.dart` shows an error banner but the "Rescan" button (line 481) clears both `_results` and `_image`, forcing the user to re-take the photo. If the failure was transient (timeout, 503), the image is lost.

**Action:** Add a "Retry" button that keeps `_image` and re-triggers `_aiService.analyzeImage(_image!)` without requiring a new photo. Only clear the image on explicit "Rescan" action.

---

### ERR-05 — Barcode lookup has no retry and swallows all exceptions under one message
**Priority: P2 | Effort: S**

`lib/screens/scan/barcode_scan_screen.dart` lines 87–95: the `catch (_)` shows "Network error. Check your connection." for every failure including JSON parse errors, timeout, and HTTP errors. The timeout is 10 seconds (`food_database_service.dart` line 39) with no user feedback while waiting.

**Action:** Show a spinner with elapsed time after 3 seconds. Distinguish network errors from parse/API errors in the user message. Add a retry button that re-calls `_lookupBarcode(_scannedBarcode!)`.

---

### ERR-06 — `DiaryProvider.addMeal` does not handle Firestore write failures
**Priority: P1 | Effort: S**

`lib/providers/diary_provider.dart` `addMeal()` (lines 42–53) awaits `_db.saveMealLog(meal)` with no try/catch. If Firestore rejects the write (e.g., security rules not yet deployed, offline cache full), the `loadMeals` call still runs, potentially showing stale data with no indication the save failed. The scan is counted against the quota (`incrementScanCount` at line 98 in `scan_screen.dart`) before the diary save occurs, so a failure deducts a scan without adding the meal.

**Action:** Wrap the `saveMealLog` call in try/catch. Only call `incrementScanCount` after a successful save. Show a SnackBar on failure.

---

### ERR-07 — `_AppEntry` shows loading spinner indefinitely if auth hangs
**Priority: P2 | Effort: S**

`lib/main.dart` calls `AuthService.getOrCreateUserId()` in the background (fire-and-forget, line 36). If this never completes (e.g., Firebase unreachable), `UserProvider.load()` may call `DatabaseService.getUser()` using `AuthService.currentUserId`, which returns `'local_user'` (the fallback), causing data to be read from the wrong document. There is no timeout on the auth call.

**Action:** Add a 10-second timeout to `AuthService.getOrCreateUserId()`. If it expires, propagate the error to `UserProvider` so the UI can display a "Try Again" prompt rather than spinning indefinitely.

### UI-06 — Streak calculation is broken — counts from index 6 backwards and stops at first zero
**Priority: P2 | Effort: S**

`lib/screens/dashboard/dashboard_screen.dart` `_calculateStreak()` (lines 64–73) iterates `for (int i = 6; i >= 0; i--)` but `_weeklyData` is indexed 0–6 where 6 is today. The loop starts at index 6 (today) and works backward, breaking on the first zero. A user who logged food Monday–Thursday but not Friday–Saturday would have a streak of 0 (because Saturday = index 6 = 0 kcal) rather than 4. The streak also resets at midnight even if the user hasn't had time to log today.

**Action:** Fix the iteration direction. Start streak from yesterday (index 5) and count backward. Today should not break the streak — only past days with no logs should.

## 6. App Store Preparation

### STORE-01 — Android release build uses debug signing config
**Priority: P0 | Effort: M**

`android/app/build.gradle` lines 44–48: `buildTypes.release.signingConfig = signingConfigs.debug`. A production APK/AAB signed with the debug keystore cannot be uploaded to Google Play and will trigger a "debug build" rejection. It also means the SHA-1/SHA-256 used for Firebase OAuth is wrong.

**Action:** Create a production keystore. Add a `signingConfigs.release` block in `build.gradle` referencing the keystore via environment variables or a `key.properties` file (which must be in `.gitignore`). Update `buildTypes.release.signingConfig = signingConfigs.release`.

### STORE-02 — iOS GoogleService-Info.plist is missing
**Priority: P0 | Effort: S**

The `ios/` directory has no `GoogleService-Info.plist`. Without it, Firebase will crash on iOS at `Firebase.initializeApp()`. `firebase_options.dart` contains the iOS configuration, so Firebase initializes from code, but some Firebase SDK features (Crashlytics, performance monitoring) require the file to be present in the bundle.

**Action:** Download `GoogleService-Info.plist` from the Firebase console for project `ai-nutriscan-f3acb` and add it to `ios/Runner/` in Xcode (not just the filesystem — it must appear in the Xcode project file).

## 7. Firebase & Backend

### FB-01 — No Firestore composite indexes defined
**Priority: P1 | Effort: S**

`lib/services/database_service.dart` `getMealsForDate` (lines 48–51) runs a compound query on `meal_logs`: `where dateTime >= start AND where dateTime < end`. Firestore requires a composite index for queries that filter on the same field twice. Without it the query will throw `[cloud_firestore/failed-precondition] The query requires an index` at runtime for new Firestore projects.

**Action:** Create `firestore.indexes.json` at the repo root with a composite index on `users/{uid}/meal_logs` for field `dateTime (ASC)`. Deploy via `firebase deploy --only firestore:indexes`. Also add an index for `scan_events.createdAt (DESC)` used in `getScanEvents`.

---

### FB-02 — `Settings.CACHE_SIZE_UNLIMITED` may cause unbounded disk growth
**Priority: P1 | Effort: S**

`lib/main.dart` line 32: `cacheSizeBytes: Settings.CACHE_SIZE_UNLIMITED`. For a diary app that accumulates data daily, the cache will grow indefinitely. On devices with limited storage, this causes `ENOSPC` errors.

**Action:** Set a bounded cache size, e.g., `cacheSizeBytes: 50 * 1024 * 1024` (50 MB). Implement a Firestore TTL policy or Cloud Function to delete `meal_logs` older than 12 months from the server side.

---

### FB-03 — No Cloud Functions for server-side business logic
**Priority: P1 | Effort: L**

All critical business logic runs client-side: scan quota enforcement, Pro status verification, RevenueCat webhook handling. A malicious user can manipulate their own Firestore data to bypass all of these. There are no Cloud Functions in the repository.

**Action:** Implement at minimum three Cloud Functions: (1) `onRevenueCatWebhook` — receive RevenueCat subscription events and update `users/{uid}.isPro` server-side; (2) `checkScanQuota` — called before Gemini, returns 429 if daily quota exceeded; (3) `cleanupOldMeals` — scheduled function to archive/delete old diary entries.

---

### FB-04 — No Firebase Crashlytics or error reporting
**Priority: P1 | Effort: S**

`firebase_analytics` is in `pubspec.yaml` but there is no `firebase_crashlytics` dependency and no error boundary in the app. Production crashes are invisible to the developer.

**Action:** Add `firebase_crashlytics: ^4.x` to `pubspec.yaml`. In `main.dart`, wrap `runApp` with a `FlutterError.onError` handler and `PlatformDispatcher.instance.onError` handler that both call `FirebaseCrashlytics.instance.recordError(...)`. Add `firebase_performance` for network request monitoring.

---

### FB-05 — Scan event `imageUrl` stores local file path, not a remote URL
**Priority: P2 | Effort: M**

`lib/screens/scan/scan_screen.dart` line 124: `imageUrl: _image?.path` stores the local file path (e.g., `/data/user/0/com.nutriscan.ai_nutriscan/cache/CAP1234.jpg`) in the Firestore `scan_events` document. This path is meaningless on another device or after the cache is cleared, and it will be wrong for future features (scan history, support review).

**Action:** Upload the scan image to Firebase Storage under `users/{uid}/scans/{scanId}.jpg` before saving the scan event. Store the download URL. Add `firebase_storage` upload logic in `_confirm()` with appropriate storage security rules.

### FB-06 — Daily scan count never resets automatically
**Priority: P1 | Effort: M**

`UserProvider.resetDailyScanCount()` exists but is never called anywhere in the codebase. The `dailyScanCount` stored in Firestore will only ever increase. Free users permanently lose their scan quota after 3 scans.

**Action:** Check `createdAt` or add a `lastScanDate` field to `UserProfile`. In `UserProvider.load()` or `incrementScanCount()`, compare the stored date to today. If different, reset `dailyScanCount` to 0 before incrementing. Alternatively, move this logic to a Cloud Function triggered on the scan event.

## 8. Missing MVP Features

### FEAT-01 — No Google Sign-In — only anonymous auth
**Priority: P1 | Effort: M**

`lib/services/auth_service.dart` only implements `signInAnonymously()`. Anonymous users cannot recover their data if they uninstall and reinstall the app. The `google_sign_in` package is not in `pubspec.yaml`. For a health app where users log months of data, account recovery is essential.

**Action:** Add `google_sign_in` and `sign_in_with_apple` (for iOS App Store requirement) to `pubspec.yaml`. Implement a sign-in screen accessible from Settings. On sign-in, link the anonymous account to the Google credential using `FirebaseAuth.instance.currentUser.linkWithCredential()` to preserve existing diary data.

---

### FEAT-02 — RevenueCat SDK initialized nowhere
**Priority: P0 | Effort: S**

`purchases_flutter` is in `pubspec.yaml` but `Purchases.configure(...)` is never called anywhere in the codebase. The SDK must be configured at app startup before any purchase calls. Without initialization, any call to RevenueCat SDK methods will throw.

**Action:** Add `Purchases.configure(PurchasesConfiguration(key))` in `main.dart` after `Firebase.initializeApp()`. Use `Platform.isIOS` to select the correct key from `AppConstants`.

---

### FEAT-03 — No scan history screen
**Priority: P1 | Effort: M**

`lib/models/scan_event.dart` and `DatabaseService.getScanEvents()` exist but no screen displays scan history. The `scan_events` collection accumulates data that is never surfaced to users. This is listed as a Pro feature ("Full History & Trends") on the paywall.

**Action:** Create `lib/screens/history/scan_history_screen.dart`. Add a "History" tab or settings entry. Display scan events with date, food names from `aiResponse`, and link to the diary entry for that day.

---

### FEAT-04 — No data export / backup
**Priority: P2 | Effort: M**

Users have no way to export their diary data. For a health tracking app, the inability to export is a common App Store review complaint and a potential GDPR/privacy law issue (users have the right to data portability).

**Action:** Add an "Export Data" option in Settings that generates a CSV or JSON of all `meal_logs` for the past year and shares it via the system share sheet using the `share_plus` package.

---

### FEAT-05 — No water / hydration tracking
**Priority: P2 | Effort: M**

The PRD context mentions CalCounti as a comprehensive tracker, but there is no water intake tracking feature. This is a standard expectation in the nutrition tracker category and is listed as a differentiator in many competing apps.

**Action:** Add a `waterIntakeMl` field to the daily diary. Add a quick-log widget to the diary header. Store daily water intake in a separate Firestore document or as a field in a daily summary document.

---

### FEAT-06 — No notification / reminder system
**Priority: P2 | Effort: M**

There is no push notification or local notification capability. Users who forget to log meals receive no reminder. Engagement and retention for habit-forming apps are significantly improved by daily logging reminders.

**Action:** Add `flutter_local_notifications` to `pubspec.yaml`. Add a notification settings section in `SettingsScreen` allowing users to set a daily logging reminder time. Schedule a local notification using the `timezone` package.

---

### FEAT-07 — Barcode scan does not trigger quota check or paywall
**Priority: P1 | Effort: S**

`lib/screens/scan/barcode_scan_screen.dart` `_lookupBarcode()` never checks `user.canScan` and never calls `incrementScanCount()`. A free user who has exhausted their AI scan quota can still add unlimited items via barcode with no restriction. This is inconsistent with the paywall model described on the paywall screen, which lists "Unlimited AI Scans" as a Pro benefit — implying barcode scans are separate — but this should be explicitly designed, not accidental.

**Action:** Decide whether barcode scans count against the free quota. If yes, add the `canScan` check and `incrementScanCount` call in `_addToDiary()`. If no, document this explicitly and ensure it is consistent with RevenueCat entitlement definitions.

## 9. DevOps & CI/CD

### DEVOPS-01 — No `.gitignore` entries for sensitive files
**Priority: P0 | Effort: S**

The standard Flutter `.gitignore` does not include `google-services.json`, `GoogleService-Info.plist`, `key.properties`, or `*.keystore`. Verify the current `.gitignore` explicitly excludes all of these files. If they have already been committed, rotate all keys and use `git filter-repo` to remove them from history.

**Action:** Add to `.gitignore`:
```
android/app/google-services.json
ios/Runner/GoogleService-Info.plist
android/key.properties
*.keystore
*.jks
```

---

### DEVOPS-02 — No CI pipeline exists
**Priority: P1 | Effort: M**

There is no GitHub Actions, Bitrise, or Codemagic configuration in the repository. Every build, test run, and release is manual. This means: (1) tests may not be run before merges; (2) release builds are built on developer machines with developer credentials; (3) there is no automated build number incrementing.

**Action:** Create `.github/workflows/ci.yml` with steps: `flutter test`, `flutter analyze`, and on tag push: `flutter build apk --release` and `flutter build ipa --release`. Store signing keys and Firebase config files as GitHub Actions secrets. Generate build numbers from `GITHUB_RUN_NUMBER`.

---

### DEVOPS-03 — `flutter analyze` produces warnings that should be CI-gated
**Priority: P1 | Effort: S**

`analysis_options.yaml` exists but its contents were not reviewed. Given the codebase patterns (use of `dynamic` in `dashboard_screen.dart` lines 428 and 437 for `_activityLabel` and `_goalLabel`, unchecked `as` casts in `database_service.dart` line 32), there are likely lint warnings that would be caught by `flutter analyze --fatal-warnings`.

**Action:** Run `flutter analyze` and fix all warnings. Enable `--fatal-warnings` in CI so new warnings block merges. Strengthen `analysis_options.yaml` to enable `avoid_dynamic_calls`, `cast_nullable_to_non_nullable`, and `unnecessary_null_checks` rules.

---

### DEVOPS-04 — No environment separation (dev / staging / prod)
**Priority: P1 | Effort: M**

There is a single Firebase project (`ai-nutriscan-f3acb`) used for all development and any future production traffic. A developer testing AI scans will generate real Firestore data in the same database as production users. API key quotas are shared.

**Action:** Create a separate Firebase project for development (`ai-nutriscan-dev`). Use `flutter run --dart-define=ENV=dev` to switch between Firebase configurations. Store separate `google-services-dev.json` and `google-services-prod.json` and select them in the build script.

---

### DEVOPS-05 — No changelog or release notes infrastructure
**Priority: P2 | Effort: S**

There is no `CHANGELOG.md`, no Git tagging convention, and no `fastlane` or `supply` configuration for automated App Store / Google Play release note submission.

**Action:** Adopt conventional commits (`feat:`, `fix:`, `chore:`). Add a `CHANGELOG.md` following Keep a Changelog format. Configure `fastlane` with `deliver` (iOS) and `supply` (Android) for automated metadata and release note uploads.
